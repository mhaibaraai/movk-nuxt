## Usage

Use the auto-imported `useApiFetch` composable for API requests. Built on top of Nuxt [`useFetch`](https://nuxt.com/docs/api/composables/use-fetch), it provides automatic authentication, data unwrapping, business status code checking and unified error handling.

```vue
<script setup lang="ts">
interface User {
  id: number
  name: string
  email: string
}

// Basic usage (automatically unwraps the data field)
const { data, pending, error, refresh } = await useApiFetch<User[]>('/users')

// POST request
const { data } = await useApiFetch('/users', {
  method: 'POST',
  body: { name: 'test', email: 'test@example.com' }
})

// Use a different endpoint
const { data } = await useApiFetch('/users', { endpoint: 'v2' })
</script>
```

- `useApiFetch` inherits all features of Nuxt [`useFetch`](https://nuxt.com/docs/api/composables/use-fetch) while providing additional API integration.
- Automatically retrieves the token from the session and adds it to request headers (via the `$api` instance).
- Automatically checks business status codes and throws errors.
- Built-in Toast notifications with customizable configuration.

## Data Request Patterns

Movk Nuxt provides three data request composables with identical signatures — choose based on execution timing:

| Composable | Execution | Blocks Navigation | Use Case |
| --- | --- | --- | --- |
| [`useApiFetch`](/docs/api/use-api-fetch) | SSR + CSR | Yes | Above-the-fold core data |
| [`useLazyApiFetch`](/docs/api/use-lazy-api-fetch) | SSR + CSR | No | Secondary / non-above-the-fold data |
| [`useClientApiFetch`](/docs/api/use-client-api-fetch) | CSR only | No | Non-SEO-sensitive data |

## Data Unwrapping and Transformation

`useApiFetch` already receives the business data unwrapped by [`$api`](/docs/api/api-plugin#data-unwrapping-and-business-validation); the generic `T` directly declares that data type. `transform` receives the unwrapped data for a secondary transformation:

```ts
// Generic T directly declares the business data type (the type of the data field)
const { data } = await useApiFetch<User>('/user')
// data.value = { id: 1, name: 'test' }

// Combine with transform for a secondary transformation
const { data } = await useApiFetch<{ content: User[] }, SelectItem[]>('/users', {
  transform: ({ content }) => content.map(u => ({ label: u.name, value: u.id }))
})
```

## Business Status Code Checking

The `$api` interceptor automatically checks the business status code and throws an `ApiError` (see [Error Classification](#error-classification) below) when the code is not in `successCodes`. Pass `skipBusinessCheck: true` to skip validation (no `ApiError` is thrown), but the `dataKey` field is still unwrapped:

```ts
const { data } = await useApiFetch('/external', {
  skipBusinessCheck: true
})
```

::tip{icon="i-lucide-arrow-right" to="/docs/api/api-plugin#data-unwrapping-and-business-validation"}
See interactive examples for business validation and `skipBusinessCheck` on the `$api` page

## Error Classification

The module only produces one custom error type — `ApiError` (business errors) — all others are ofetch's native `FetchError`. Distinguish scenarios with `isBusinessError` and `statusCode`. Select an error type in the dropdown to observe the shape of `error.value`:

- `business`: HTTP 200 but `code` not in `successCodes` → `ApiError`, `isBusinessError` is true.
- `http-400` / `http-500`: Native `FetchError` with `statusCode` set to the corresponding code.
- `http-422`: `FetchError` whose `data` field carries additional server information.
- `network`: Connection forcibly cut → `FetchError` with no `statusCode`.

## Toast Notifications

The `toast` option supports per-request configuration: `false` to disable all, `{ success: false }` to keep only errors, `successMessage` / `errorMessage` for quick text, or a full set of [`Toast`](https://ui.nuxt.com/components/toast) props (`color`, `icon`, `duration`, etc.):

```ts
// Disable Toast / keep error only / quick text
await useApiFetch('/users', { toast: false })
await useApiFetch('/users', { toast: { success: false } })
await useApiFetch('/users', {
  toast: { successMessage: 'Created!', errorMessage: 'Failed, please retry' }
})

// Full Toast props
await useApiFetch('/users', {
  toast: {
    success: { title: 'Created', color: 'success', icon: 'i-lucide-circle-check' },
    error: { title: 'Failed', color: 'error', duration: 5000 }
  }
})
```

::tip{icon="i-lucide-arrow-right" to="/docs/api/api-plugin#toast-notifications"}
See interactive examples for five Toast modes on the `$api` page

## API

### useApiFetch()

`useApiFetch<T = unknown, DataT = T>(url: string | (() => string), options?: UseApiFetchOptions<T, DataT>): UseApiFetchReturn<DataT>`{lang="ts-type"}

Creates an API request.

#### Parameters

::field-group
  ::field{name="url" type="string | (() => string)" required}
  Request URL or a function returning the URL. Supports reactive URLs.
  ::

  ::field{name="options" type="UseApiFetchOptions<T, DataT>"}
  Request configuration options. Supports all Nuxt `useFetch` options plus additional API integration options.

    ::collapsible

  **API Integration Options**

      ::field-group
        ::field{name="endpoint" type="string"}
  Use the specified endpoint configuration. Defaults to the default endpoint of `$api`.
        ::

        ::field{name="toast" type="RequestToastOptions | false"}
  Toast notification configuration. Set to `false` to disable Toast.

          ::collapsible

            ::field-group
              ::field{name="success" type="Partial<Toast> | false"}
  Success notification configuration. Can be a full Toast props object or `false` to disable success notifications.
              ::

              ::field{name="error" type="Partial<Toast> | false"}
  Error notification configuration. Can be a full Toast props object or `false` to disable error notifications.
              ::

              ::field{name="successMessage" type="string"}
  Quick text for success notifications. Equivalent to `success: { title: '...' }`.
              ::

              ::field{name="errorMessage" type="string"}
  Quick text for error notifications. Equivalent to `error: { title: '...' }`.
              ::
            ::
          ::
        ::

        ::field{name="skipBusinessCheck" type="boolean"}
  Skip business status code checking (no `ApiError` is thrown), but the `dataKey` field is still unwrapped. Defaults to `false`.
        ::

        ::field{name="skipUnwrap" type="boolean"}
  Skip data unwrapping. When `true`, returns the full `code/message/data` envelope. Orthogonal to `skipBusinessCheck`. Defaults to `false`.
        ::
      ::

  **Request Options** (inherited from `useFetch`)

      ::field-group
        ::field{name="method" type="'GET' | 'HEAD' | 'PATCH' | 'POST' | 'PUT' | 'DELETE' | 'CONNECT' | 'OPTIONS' | 'TRACE'"}
  HTTP method. Defaults to `'GET'`.
        ::

        ::field{name="query" type="Record<string, any>"}
  URL query parameters, automatically serialized into the URL.
        ::

        ::field{name="params" type="Record<string, any>"}
  Alias for `query`.
        ::

        ::field{name="body" type="RequestInit['body'] | Record<string, any>"}
  Request body. Automatically serialized to JSON for POST/PUT/PATCH requests.
        ::

        ::field{name="headers" type="Record<string, string> | [key: string, value: string][] | Headers"}
  Request headers.
        ::

        ::field{name="baseURL" type="string"}
  Base URL. Usually configured via the `endpoint` option; no need to set manually.
        ::
      ::

  **Response Handling**

      ::field-group
        ::field{name="transform" type="(input: T) => DataT | Promise<DataT>"}
  Function to transform response data. Receives the unwrapped business data and returns the final data.
        ::

        ::field{name="pick" type="string[]"}
  Select only the specified fields from the response data.
        ::

        ::field{name="default" type="() => DataT"}
  Factory function to set a default value.
        ::
      ::

  **Execution Options**

      ::field-group
        ::field{name="immediate" type="boolean"}
  Whether to execute the request immediately. Set to `false` to call `execute()` manually. Defaults to `true`.
        ::

        ::field{name="lazy" type="boolean"}
  Whether to use lazy mode. In lazy mode, client-side navigation is not blocked and data loads in the background. Defaults to `false`.
        ::

        ::field{name="server" type="boolean"}
  Whether to execute the request on the server. Defaults to `true`.
        ::

        ::field{name="watch" type="MultiWatchSources"}
  Reactive sources to watch; the request re-fires automatically on change. Can be a ref, reactive object or array.
        ::

        ::field{name="deep" type="boolean"}
  Whether to deep-watch the `watch` objects. Defaults to `true`.
        ::
      ::

  **Caching and Deduplication**

      ::field-group
        ::field{name="key" type="string"}
  Custom cache key. Used for sharing data between requests or deduplication.
        ::

        ::field{name="dedupe" type="'cancel' | 'defer'"}
  Strategy for handling duplicate requests. Defaults to `cancel`.

  - `'cancel'` — Cancel the pending request
  - `'defer'` — Do not initiate a new request
        ::

        ::field{name="getCachedData" type="(key: string) => DataT"}
  Function to retrieve data from cache. The return value is used as the initial `data`.
        ::
      ::

  **Request Hooks**

      ::field-group

        ::field{name="onRequest" type="(context: FetchContext) => Promise<void> | void"}
  Hook called before the request is sent. Merged with built-in hooks.

          ::collapsible
          ```ts
          onRequest({ request, options }) {
            console.log('Sending request:', request.url)
          }
          ```
          ::
        ::

        ::field{name="onRequestError" type="(context: FetchContext & { error: Error }) => Promise<void> | void"}
  Hook called on request error. Merged with built-in hooks.

          ::collapsible
          ```ts
          onRequestError({ request, error }) {
            console.error('Request error:', error)
          }
          ```
          ::
        ::

        ::field{name="onResponse" type="(context: FetchContext & { response: Response }) => Promise<void> | void"}
  Hook called when a response is received. Built-in hooks run first, so `response._data` is already the unwrapped business data (unless `skipUnwrap: true`); only fires on business success. Return values are ignored by ofetch — use `transform` for output transformations.

          ::collapsible
          ```ts
          onResponse({ response }) {
            console.log('Unwrapped data:', response._data)
          }
          ```
          ::
        ::

        ::field{name="onResponseError" type="(context: FetchContext & { response: Response }) => Promise<void> | void"}
  Hook called on response error (4xx/5xx). Merged with built-in hooks.

          ::collapsible
          ```ts
          onResponseError({ response }) {
            console.error('Response error:', response.status)
          }
          ```
          ::
        ::
      ::
    ::
  ::
::

#### Type Parameters

::field-group
  ::field{name="T" type="type"}
  Business data type (already automatically unwrapped by `$api`). This is the data type of the `data` field in the API response.
  ::

  ::field{name="DataT" type="type"}
  The final type after `transform` conversion. Defaults to `T`.
  ::
::

#### Returns

Returns the response object of [`useFetch`](https://nuxt.com/docs/api/composables/use-fetch#return-values), including:

::field-group
  ::field{name="data" type="Ref<DataT | null>"}
  Response data (unwrapped and transformed). Initially `null`; updated to the actual data after a successful request.
  ::

  ::field{name="error" type="Ref<FetchError | ApiError | null>"}
  Error object. Network errors are `FetchError`; business status code errors are `ApiError` (with `statusCode`, `response` and `isBusinessError` properties).
  ::

  ::field{name="status" type="Ref<'idle' | 'pending' | 'success' | 'error'>"}
  Request status.

- `'idle'` — Not yet started (only when `immediate: false`)
- `'pending'` — Request in progress
- `'success'` — Request succeeded
- `'error'` — Request failed
  ::

  ::field{name="pending" type="Ref<boolean>"}
  Whether a request is in progress. Equivalent to `status.value === 'pending'`.
  ::

  ::field{name="refresh" type="(opts?: { dedupe?: boolean }) => Promise<void>"}
  Refresh data by re-executing the request. Alias: `execute`.

    ::collapsible
    ```ts
    // Refresh data
    await refresh()

    // Skip deduplication check and force refresh
    await refresh({ dedupe: false })
    ```
    ::
  ::

  ::field{name="execute" type="(opts?: { dedupe?: boolean }) => Promise<void>"}
  Alias for `refresh`. Manually execute the request (commonly used with `immediate: false`).
  ::

  ::field{name="clear" type="() => void"}
  Clear state: sets `data` to `null`, `error` to `null` and `status` to `'idle'`.
  ::
::
